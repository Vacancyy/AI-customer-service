## 一、评估方法说明

### 1.1 什么是RAGAS

RAGAS（Retrieval Augmented Generation Assessment）是业界最广泛使用的RAG系统评估框架，由explodinggradients开源维护，被Google、Microsoft、AWS等企业采用。与传统"LLM打分"方式不同，RAGAS对RAG的每个环节分别评估，能定位问题出在检索阶段还是生成阶段。

### 1.2 评估指标详解

#### Faithfulness（忠实度）

**评估什么**：AI回答中的每一个事实声明，是否都能被检索到的知识库内容所支撑。

**怎么算**：
1. 将AI回答拆分为多个独立的事实声明（claim）
2. 对每个claim，判断是否能从检索上下文中找到依据
3. 忠实度 = 有依据的claims数 / 总claims数

**为什么重要**：这是RAG系统最关键的指标——回答"看起来对但实际上编造了细节"是最危险的问题，尤其在保险场景可能导致用户误解保障内容。

**举例**：
```
用户问：理赔需要哪些材料
AI回答：需要住院发票、出院小结、身份证
知识库：需要住院发票、出院小结、身份证、银行卡、医保卡

→ AI遗漏了"银行卡、医保卡"，但没编造，忠实度仍然高
→ 如果AI说"需要支付宝截图"，知识库里没有，忠实度下降
```

#### Response Relevancy（回答相关性）

**评估什么**：AI回答是否聚焦于用户提出的问题，而不是答非所问或过度展开。

**怎么算**：
1. 用LLM从AI回答中反向生成可能的问题
2. 计算反向生成的问题与原问题的语义相似度（基于embedding）
3. 相关性 = 相似度的加权和

**为什么重要**：用户问"保费多少"，AI回答了一大段保障范围和理赔流程，虽然内容正确但不够切题，用户体验差。

**举例**：
```
用户问：保费多少钱
AI回答：保费99元/150元两款。南京宁惠保保障范围包括...

→ 前半句切题，后半句多余，相关性降低
```

#### Context Precision（检索精度）

**评估什么**：检索到的Top-K条知识库内容中，有多少是真正与问题相关的。

**怎么算**：
1. 对每条检索结果，判断它是否包含回答问题所需的信息
2. 精度 = 相关结果在排序中的加权得分

**为什么重要**：如果检索了3条，只有1条相关，不相关的2条会干扰LLM生成，增加"串答"风险。

**举例**：
```
用户问：如何退保
检索Top-3：[退保流程✓, 退保费用计算✓, 理赔流程✗]

→ 第3条不相关，精度降低
→ 不相关内容可能导致AI回答中混入理赔相关信息
```

#### Factual Correctness（事实正确性）

**评估什么**：AI回答中的事实是否与知识库标准答案一致。

**怎么算**：
1. 将AI回答和标准答案分别拆分为独立事实集合
2. 计算事实级别的Precision和Recall
3. F1 = 2 × P × R / (P + R)

**为什么重要**：忠实度保证"不编造"，但回答可能遗漏关键信息。事实正确性同时检查"有没有错"和"有没有漏"。

**举例**：
```
标准答案：免赔额1.5万(基础版) / 1.45万(升级版)
AI回答：免赔额1.5万

→ 没有编造，忠实度OK
→ 但遗漏了升级版，事实正确性下降
```

### 1.3 与之前评估方式的对比

| 维度 | 之前（rag_eval.py） | 本次（RAGAS） |
|------|---------------------|---------------|
| 评估框架 | 自定义prompt让LLM打分 | 业界标准RAGAS框架 |
| 评分粒度 | 4个0-10分（笼统） | 4个0-1分（细粒度） |
| 忠实度检测 | "回答是否忠于知识库？" | 逐条claim验证有无依据 |
| 相关性检测 | "回答是否相关？" | 反向生成问题计算embedding相似度 |
| 检索质量 | 不评估 | 独立评估检索精度 |
| 事实完整性 | 笼统打分 | 事实级Precision/Recall/F1 |
| 可复现性 | 依赖prompt措辞 | 标准化指标，可跨系统对比 |
| 评判模型 | Qwen-Max | Qwen-Max（相同） |

### 2.3 问题样本分析

**最差样本（第1题）：医疗费用发票原价丢失了怎么办**
- 忠实度0.70：AI在回答末尾添加了"这个问题需要咨询人工客服"，但知识库原文并未建议转人工，属于无依据的补充
- 相关性0.00：RAGAS的反向问题生成判断该回答偏离了原问题焦点（原问"丢失了怎么办"，AI回答侧重"发票的作用"而非"丢失后的补救措施"）
- 事实正确性0.55：遗漏了关键信息

**薄弱样本（第7题）：住院手术费用是否可以报销**
- 事实正确性0.45：知识库中有详细的报销条件和比例说明，AI回答过于简略，遗漏了既往症/非既往症的赔付比例差异

**薄弱样本（第10题）：门诊费用是否可以报销**
- 检索精度0.00：检索到的3条知识库内容都不包含直接回答该问题的关键信息，说明该问题在知识库中的覆盖不足
- 事实正确性0.55：因检索源头不准，回答也不够精确

## 三、问题诊断

### 3.1 系统优势

1. **防幻觉效果显著（忠实度0.97）**：prompt中的"禁止编造"约束有效，17题中14题忠实度为1.0
2. **检索质量高（精度0.92）**：222条知识库的语义检索Top-3精准命中，大部分题目的检索精度为1.0
3. **简单问题回答准确**：对于"什么时候生效""怎么投保""犹豫期多久"等明确问题，4项指标全部满分

### 3.2 需要改进的问题

| 问题 | 根因 | 影响 |
|------|------|------|
| **回答相关性偏低（0.82）** | 8B模型倾向于过度展开，把相关但不必要的信息也写进去 | 用户需要的信息被淹没，阅读体验差 |
| **复杂问题遗漏细节** | 模型对"基础版/升级版""既往症/非既往症"的区分不够完整 | 用户可能基于不完整信息做决策 |
| **知识库覆盖不足** | "门诊费用报销"等问题的检索精度为0，知识库中没有直接对应的QA | 无法回答或回答不准确 |
| **转人工逻辑简单** | 回答不了时统一加"请拨打4000040181"，有时出现在已有回答之后，逻辑矛盾 | 第1题因此忠实度下降 |

## 四、优化建议

### 4.1 回答相关性优化（优先级：高）

**目标**：将回答相关性从0.82提升至0.90+

- 调整prompt：增加"只回答用户问的问题，不要主动扩展"的约束
- 控制回答长度：将max_tokens从600降低至400，减少过度展开的空间
- 增加示例：在prompt中加入"好的回答"和"过度展开的回答"的对比示例

### 4.2 事实完整性优化（优先级：高）

**目标**：将事实正确性从0.87提升至0.93+

- prompt增加"如果知识库中有基础版和升级版两套数据，必须完整列出两版的数字"（已有但需强化）
- 知识库中拆分长答案：将"免赔额具体金额"这类包含多版本信息的答案，拆分为更结构化的格式

### 4.3 知识库补充（优先级：中）

- 补充"门诊费用报销条件"的直接QA
- 七期条款更新后替换六期数据
- 清理知识库中的特定客户信息（如"4.16的清单""72506.58"等），避免出现在回答中

### 4.4 转人工逻辑优化（优先级：中）

- 当confident回答时（相似度>0.8），不加转人工提示
- 当不自信时（相似度0.3-0.5），主动建议咨询人工
- 避免在有效回答末尾硬加"请拨打人工客服"
